Git Branch

Use branches to isolate work and keep main stable.

What Is a Branch?

Isolate work without breaking stable code

A Git branch is a lightweight pointer to a commit. It lets you work on a feature, experiment, or bug fix in isolation, without interfering with the main line of development.

Think of branches as parallel timelines of the same project. Each branch has its own commit history, and you can switch between them as your priorities change.

Why Branches Are Useful

Handle context switching with confidence

Imagine you are building a large feature and have several unfinished changes. Suddenly, a critical production bug is reported and must be fixed immediately.

Without branches, you would either mix unrelated changes together or lose track of partial work. With branches, you can keep your in-progress feature safely on one branch, switch to another branch for the urgent bug, ship the fix, and then return exactly where you left off.

This clean separation makes collaboration easier for teams and keeps pull requests focused, reviewable, and lower risk.

How Branches Work in Git

Pointers to snapshots

Every commit in Git is a snapshot of your project. A branch is simply a named pointer to one of these snapshots.

When you create a new branch, Git does not copy all files into a second project. It creates a new pointer from your current commit. As you add commits on that branch, the pointer moves forward independently from other branches.

Historically, many repositories used master as the default branch. Today, most repositories use main. The behavior is identical either way: it is just the primary branch name.

How a new branch diverges from the parent branch: both branches start at the same commit, then each pointer moves as new commits are added.

Branches from Other Branches

Not only from main

You can create a branch from any existing branch, not just from main. This is useful when work depends on another in-progress change.

Example: if feature/api-refactor is still open, you can branch from it into feature/api-refactor-logging to build a smaller related improvement without waiting.

Understanding git branch

List and read branch output

Running git branch lists your local branches and marks your current branch with an asterisk (*).

CommandTypical output
git branch* main
feature-login
bug-fix

In this example, you are currently working on main. The other listed branches exist, but are inactive until you switch to them.

Creating vs Switching

A common beginner mistake

Creating a branch and switching to a branch are two separate actions.

If you run git branch feature-payment, Git creates the branch but keeps you on your current branch.

To start working on the new branch, run git checkout feature-payment (or git switch feature-payment).

Important: Creating does not mean switching.

Real-Life Workflow Example

Building safely on a branch

Suppose your main branch is a stable website and you want to add login functionality:

  • Create branch: git branch login-feature
  • Switch branch: git checkout login-feature
  • Build and test freely on login-feature

Your primary branch remains stable while you experiment and iterate.

Shortcut Commands

Create and switch in one step

Instead of running two commands:

  • git branch new-branch
  • git checkout new-branch

Use one shortcut:

git checkout -b new-branch

This creates the branch and switches to it immediately. The modern equivalent is git switch -c new-branch.

Quick Summary

What to remember

  • git branch lists branches.
  • git branch <name> creates a branch.
  • git checkout <name> (or git switch <name>) switches branch.
  • The asterisk (*) shows your current branch.
  • Creating a branch does not automatically switch to it.

Best Practices

Keep history easy to review

  • Use short-lived, purpose-driven branches.
  • Write clear names, for example fix/login-timeout or feature/billing-export.
  • Sync regularly with latest main to reduce merge conflicts later.
  • Open small pull requests so teammates can review quickly and safely.
