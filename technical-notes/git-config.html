Git

Git Config

Learn how to configure Git as a reliable daily tool, not just set your name and email once. This guide covers config scope precedence, practical defaults, credential storage, and troubleshooting so your commits are consistent across personal, work, and open-source repositories.

Installing Git

Windows, macOS, and Linux

  • Install Git from the official installer for your operating system.
  • After installation, use your terminal to run Git commands locally.
  • Verify installation with:
git --version

If installed correctly, the command prints your Git version (example: git version 2.48.1).

How git config Works

Precedence and where values come from

Level Scope When to use
--local Current repository only Project-specific identity or settings
--global Current user on this machine Default identity for all your repositories
--system All users on the machine Machine-wide policy (usually admin-managed)

When the same key exists at multiple levels, Git uses the most specific one:

local repository value > global user value > system value

See every active key with file origin:

git config --list --show-origin --show-scope

First-Time Baseline Setup

Strong defaults for most developers

Set global identity and base behavior once per machine:

git config --global user.name "Shishir Bansal"
git config --global user.email "shishir.bansal@example.com"
git config --global init.defaultBranch main
git config --global core.editor "code --wait"
git config --global pull.rebase false

Verify all important baseline values:

git config user.email
git config user.name
git config init.defaultBranch
git config core.editor
git config pull.rebase

Identity Strategy

Personal, company, and privacy-safe options

  • Use your real name so commit history stays readable for teams.
  • Use an email tied to the account where you push code, especially on GitHub.
  • For public repos, use a no-reply address if you do not want to expose personal email.
  • Use local overrides for client or company repos that require a different identity.

Local Override Example

One repository, different author profile

git config --local user.email "work.profile@example.com"
git config --local user.name "Shishir Bansal (Work)"

Confirm exactly what this repository will use:

git config --local --list
git config --show-origin user.email
git config --show-origin user.name

Local values override global values only in the current repository.

Credential Setup

Avoid typing tokens repeatedly

Configure a credential helper so Git stores authentication securely:

Platform Recommended helper Command
Windows Git Credential Manager git config --global credential.helper manager-core
macOS Keychain git config --global credential.helper osxkeychain
Linux Credential cache (short-term) git config --global credential.helper cache

If your organization standardizes on Git Credential Manager, follow that policy across all platforms.

Quality-of-Life Defaults

Small settings that improve daily workflow

git config --global fetch.prune true
git config --global rebase.autoStash true
git config --global core.autocrlf input
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --decorate --all"
  • fetch.prune removes deleted remote-tracking branches when you fetch.
  • rebase.autoStash helps when rebasing with uncommitted working changes.
  • core.autocrlf input avoids accidental line-ending noise on macOS/Linux.
  • Aliases reduce repetitive typing for common commands.

Troubleshooting

Common config issues and quick fixes

Symptom Command to diagnose/fix
Author identity unknown git config --global user.name "Your Name" && git config --global user.email "you@example.com"
Wrong email on commits git config --show-origin user.email
Unexpected merge/rebase behavior git config --show-origin pull.rebase
Credential prompts on every push git config --global credential.helper

Creating a New Project with Git

Initialize a repository and start tracking files

In this guide, you will learn how to set up a new project using Git for version control. You will create a project directory, initialize a Git repository, add files for tracking, and check repository status. By the end, you will understand essential commands like git init, git add, and git status.

1. Create a New Project Directory

Use the command below to create a new working directory for your project:

mkdir test_project
cd test_project

After this step, your working directory is test_project and it is empty.

2. Create a File in the Project

Create a simple text file and confirm it exists:

touch file1.txt
ls

You should see file1.txt in the output.

3. Initialize a Git Repository

git init

What git init does:

  • Initializes an empty Git repository in your project.
  • Creates a hidden .git directory.
  • Stores Git metadata and version history in that .git folder.

Running git init does not automatically track files.

4. Add Files to Git Tracking

git add .

What this command does:

  • . means all files in the current directory.
  • Stages files so they are ready for commit.
  • Begins tracking the added files.

5. Check Repository Status

git status

Why git status matters:

  • Shows untracked files (not added yet).
  • Shows staged files (ready to commit).
  • Shows modified files (changed after last commit).
  • Displays information only and does not change your files.

Example workflow:

  • Before git add ., file1.txt appears as untracked.
  • After git add ., file1.txt appears as staged.

Conclusion

  • Created a new project directory.
  • Added a file to the project.
  • Initialized a Git repository.
  • Started tracking files with Git.
  • Checked repository status.

These steps are the foundation for using Git effectively in any project.

Git Commit

Create your first commit and understand HEAD

In this section, you will learn how to create your first Git commit and understand how Git tracks your project's history using commits. A commit is one of the most important concepts in Git, as it represents a saved snapshot of your project at a specific point in time.

1. What Is a Commit?

A commit is a snapshot of your entire project at a given moment. Each commit records the state of all tracked files, allowing you to revisit or restore previous versions whenever needed.

Git maintains your project history as a sequence (or chain) of commits.

Understanding HEAD:

  • HEAD represents the most recent commit in your repository.
  • Every new commit becomes the new HEAD.
  • Each commit is linked to the previous one, forming a timeline of changes.

This structure allows Git to efficiently track and manage your project's evolution.

2. The git commit Command

Once you have staged your files using git add, you are ready to create a commit.

Basic syntax:

git commit -m "Your commit message"

Explanation:

  • git commit creates a new commit.
  • -m stands for message.
  • "Your commit message" is a short description of the changes made.

3. Importance of Commit Messages

The commit message explains what changes were made in that snapshot.

Best practices for commit messages:

  • Keep messages clear and descriptive.
  • Use meaningful wording.
  • Describe what changed and why.

Good examples:

initial commit
added login functionality
fixed bug in payment processing

Avoid:

commit 1
update
changes

Well-written commit messages make project history easier to understand and collaborate on.

4. Creating Your First Commit

Before creating a commit, check repository status:

git status

This ensures the correct files are staged and ready to be committed.

Now create your first commit:

git commit -m "initial commit"

5. What Happens After a Commit?

  • All staged files become part of the commit.
  • Git saves a snapshot of your project.
  • The commit becomes the new HEAD of the repository.
  • Your working directory is now clean (no pending staged changes).

To verify this, run:

git status

You should see a message indicating that there is nothing left to commit.

6. Summary

  • Created your first Git commit.
  • Captured a snapshot of your project.
  • Learned how Git tracks history using commits and HEAD.
  • Understood the importance of meaningful commit messages.

You have now taken a major step in mastering Git version control.

Quick Checklist

Before committing in a new environment

  • Git is installed and verified with git --version.
  • Global user.name and user.email are configured.
  • Your branch and pull defaults match your team workflow.
  • Credential helper is configured for your operating system.
  • Any project requiring different identity has local overrides.
  • You can inspect config precedence with --show-origin.
